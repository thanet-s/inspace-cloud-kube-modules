## One-or-three control-plane topology

`InSpaceCluster.spec.controlPlane.replicas` now accepts exactly `1` or `3` and
is immutable after creation. A single server is the low-cost topology and has
the expected API/embedded-etcd single point of failure; Karpenter application
workers remain elastic and start at zero. Three servers retain the existing
embedded-etcd HA topology. Two servers are rejected.

The bootstrap controller retains three fixed durable-ledger slots for backward
compatibility but reconciles and destroys only the configured topology. A
single-server cluster fails closed if cp1/cp2 VMs, FIPs, firewall assignments,
or durable create/delete receipts appear. Existing three-server v0.6 clusters
retain their ownership and teardown compatibility.

## Operator Ansible lifecycle

The new `deploy/` workflow is separate from the isolated-account release E2E.
It provides a gitignored operator inventory and durable cluster-scoped state,
resumable initialization, status and private-API tunnel commands, rolling RKE2
operator configuration, exact OCI chart upgrades, and owner-scoped destroy.
The committed example inventory requires an SSH username and matching private
and public key files; the private key remains on the management host.
Bootstrap-controller pull and execution explicitly select `linux/amd64`, so
the lifecycle also works from Apple Silicon Docker Desktop while InSpace and
the published controller image remain x86-64 only.

Initialization never overwrites an existing bootstrap YAML ledger. It renders
a candidate spec, pins its exact bootstrap-controller version, and rejects
drift before remote mutation. It then uses that recorded image, pins SSH host
keys, retrieves kubeconfig through the bastion, and installs the CCM, CSI,
Karpenter, default AMD EPYC NodeClass, and zero-at-rest NodePool.

Destroy requires an exact cluster-name confirmation. It refuses to proceed
while PVC, PV, or VolumeAttachment ownership remains, removes LoadBalancer
Services while CCM is healthy, drains all Karpenter NodeClaims and non-control
plane nodes, and only then removes charts and invokes the original recorded
bootstrap-controller version against its durable ledger. VPCs, manually
managed floating IPs, and unrelated account resources are outside its deletion
scope.

## Upgrade and replacement limits

The Ansible update playbook manages an operator RKE2 config fragment and rolls
at most one fixed server at a time. It blocks topology, node identity, CNI,
CIDR, token, data-directory, and system-registry overrides owned by bootstrap.
A single-server restart necessarily causes brief API downtime.

Fixed control-plane image, shape, RKE2 version, cache mode, network, VIP, and
replica count are not in-place updates. Those changes still require an
explicit destroy/recreate or future state-aware replacement lifecycle.
