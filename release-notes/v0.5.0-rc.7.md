## Karpenter firewall-assignment convergence

RC.6 live acceptance exposed an eventually consistent InSpace firewall
readback: immediately after a successful assignment, the intended firewall can
temporarily list the same VM resource twice. The provider treated that safe
intermediate observation as terminal ownership drift, rolled the VM and FIP
back correctly, and then waited for Karpenter's failed-launch replacement
cycle.

RC.7 retries only that exact duplicate-intended observation within the existing
bounded post-assignment readback window. It never repeats the assignment
mutation. A canonical single assignment may proceed; a persistent duplicate
still times out and invokes the protected rollback path. Established-state
duplicates, foreign or malformed assignments, duplicate firewall identities,
and invalid NodeLB shard or shared-ICMP policies remain immediate fail-closed
errors.

Regression coverage proves transient convergence before public-FIP
publication, persistent-duplicate rollback with zero VM/FIP residue, strict
established-state rejection, and exclusion of mixed foreign assignments from
the retry path. Full destructive RC.7 acceptance is still required; these
notes do not claim that it has passed.

## Smaller default NodeLB capacity

The shared and dedicated public NodeLB default is now 1 vCPU, 2 GiB RAM, and a
30 GiB root disk on AMD EPYC. The finite Karpenter catalog again advertises
`is-general-1c-2g`, including its numeric CPU/memory labels and the same
CPU-plus-RAM price formula as every other shape. Dedicated Services may select
that minimum explicitly; larger exact catalog shapes remain supported.

The guarded RC.7 suite leaves CPU and memory unset on its public NodeLB
Services and requires every generated NodePool, NodeClaim, and Ready Node to
prove the 1-vCPU/2-GiB shape. Traffic, mixed-protocol, firewall mutation, and
full cleanup checks therefore run on the new default rather than only testing
it as a catalog entry.

## NodeLB teardown convergence

NodeLB NodePools now request foreground deletion. Kubernetes can therefore
terminate their `blockOwnerDeletion` NodeClaims while the CCM state finalizer
keeps the NodePool and its aggregate-firewall ledger available. Once capacity
is absent, CCM deletes and proves absence of the shard firewall before releasing
its finalizer. RC.6 used background deletion, which could leave the state
finalizer waiting for dependents whose garbage collection could not begin. CCM
also upgrades a user-initiated background deletion to an exact UID-fenced
foreground request only while managed NodeClaims remain, and never re-adds the
foreground finalizer after they drain. The guarded live suite intentionally
starts background deletion on the dedicated shard, requires that same live
NodePool UID to acquire `foregroundDeletion` while its exact NodeClaim remains,
then requires ordinary Service cleanup to restore the complete account
inventory with zero residue.
