## Endpoint-local public Services

v0.6.0 adds `public-node-local`, the fifth load-balancing mode. It exposes the
floating IPs of operator-owned edge nodes only while those nodes are Ready and
host Ready local endpoints. The mode uses `externalTrafficPolicy: Local`,
preserves the real client address, and supports low TCP and UDP ports without
data-port NodePorts. Kubernetes still allocates its standard
`healthCheckNodePort` for each Local LoadBalancer Service; CCM does not expose
that port through the InSpace firewall. `publishNotReadyAddresses: true` is
rejected so an unready endpoint cannot bypass the Local readiness gate.

CCM owns only an exact private Cilium child Service and one canonical
per-Service InSpace firewall. It does not create, move, or delete the selected
VMs or FIPs. Karpenter-backed edge nodes require a `public-node-local`
NodeClass plus exact NodeClaim, NodePool, NodeClass, VM-ownership, address, and
FIP proofs. Karpenter-backed nodes also prove public-IP reservation, the
configured trusted private base firewall, and the VM's complete
base-plus-Service firewall assignment set. Manually managed nodes require an
administrator-applied protected pool label and the same providerID, VM,
address, and FIP checks.

Pool-wide protocol/port arbitration prevents duplicate frontends. Firewall
detachment and authoritative readback precede private/public status
withdrawal, and stale Service transitions block port reuse until the former
frontend is closed. Policy, ownership, readiness, or cloud drift therefore
fails closed.

The included static AMD EPYC NodePool example supports an explicit replica
count, disables expiration and voluntary disruption, isolates edge workloads
with a taint, and allows a replacement surge. Because addresses belong to
nodes, a roll can change the published FIP; use short DNS TTLs where that
matters. Stable promotion also requires a forced live NodeClaim replacement to
prove old VM/FIP cleanup and Service reconvergence on the replacement.
The guarded acceptance probes both TCP/80 and UDP/443 through the public FIP
before and after that replacement.

## Upgrade note

The feature is additive relative to v0.5.0. Existing private, paid InSpace
NLB, `public-node-shared`, and `public-node-dedicated` Services require no
migration. Stable promotion uses the exact commit whose published release
candidate passed the guarded real-cloud acceptance and zero-residue cleanup.

## RC.2 live-acceptance corrections

Destructive RC.2 acceptance reached a working cluster but exposed additional
sparse and soft-deleted InSpace readbacks plus several pre-dispatch receipt
lifecycle bugs. Stable v0.6.0 remains gated on publishing these corrections as
a new release candidate and passing the complete guarded real-cloud lifecycle,
including zero-residue cleanup. These notes do not claim that gate has passed.

InSpace VM detail and list responses may omit `network_uuid`. Bootstrap
mutation authority, CSI attachment, CCM load-balancer target/mutation
authority, and Karpenter fenced-create/cleanup authority accept that omission
only when an exact configured-VPC read independently proves one membership for
the canonical VM UUID. A present contradictory network, wrong billing account
or UUID, duplicate/missing membership, or VM with status `Deleted` still fails
closed. CCM's read-only InstancesV2 identity paths use `ListVMs` only for name
discovery, re-read the exact VM, and require the same configured-VPC membership
proof for every active VM even when `network_uuid` is present. Exact
HTTP-200 `Deleted` tombstones may omit stale VPC membership but are reported
only as `InstanceNotFound`. The same active-tombstone rejection covers all
NodeLB advertisement modes. This allows CSI attachment and load-balancer
target reconciliation against sparse live VM responses without weakening VPC
ownership.

CCM no longer lets one exact VM-detail 404, normalized missing-VM HTTP 400, or
HTTP-200 `Deleted` tombstone make `InstanceExists` return false.
ProviderID-backed absence requires the same call to prove omission from both a
valid VM inventory and the exact configured VPC's non-nil membership list. Any
read error, malformed or duplicate identity, or positive row is a split-view
error. Before providerID initialization, a name-list omission also remains an
error so cloud-node-lifecycle cannot delete a NotReady Node during the
external-CCM race.

After disk deletion, InSpace may keep returning the exact disk as HTTP 200 with
status `Deleted` while omitting it from the active disk list. CSI treats only a
canonical UUID/billing tombstone corroborated by that independent list
omission as one absence observation; its existing three spaced, durable
observations are still required. Ordinary volume reads and attachment reject
that tombstone as absent, so a terminal disk cannot be reused. Paid NLB
deletion applies the same rule to a validated `is_deleted` tombstone and the
active NLB list.

The stock Kubernetes Service controller calls `GetLoadBalancer` before
`EnsureLoadBalancerDeleted`. CCM now keeps a deleting paid-NLB Service in the
cleanup path when its exactly owned FIP is fully unassigned or a durable
receipt remains, allowing the provider to finish cleanup before Kubernetes
drops the Service finalizer. A live Service, foreign assignment, or residual
assignment metadata remains an error. All ten paid-NLB mutation paths also
CAS-clear an exact newly issued receipt after a proven local rejection before
the cloud method was called; target reconciliation does not start additive
work for a deleting Service.

Karpenter cleanup now reserves a complete bounded absence window for every
sequential VM, FIP, and firewall-relation phase, so an earlier near-timeout
phase cannot starve the final audit or strand the NodeClaim protection
finalizer. Its active discovery, adoption, and mutation-authority paths reject
HTTP-200 VM objects whose status is `Deleted`. Deletion-only recovery accepts
such a tombstone as the exact-GET negative only after immutable UUID, name,
billing, launch-ownership, and durable-anchor validation; location inventory
and configured-VPC membership must independently omit the VM for the existing
three spaced observations. Floating-IP and firewall-dependent cleanup uses the
same authority, so a committed delete that returned an HTTP error can converge
after restart without replaying `DeleteVM`.

Every NodeLB firewall create, update, delete, assignment, and unassignment path
now follows the same receipt rule as the paid NLB controller. A final ownership
or policy rejection before the SDK call, or the SDK's typed local
`ErrMutationBlocked`, resets only the exact CAS-winning issue through a bounded
detached Kubernetes write. HTTP responses, timeouts, disconnects, malformed
successes, and redirects remain post-dispatch ambiguous and retain the receipt
until authoritative readback converges. Shard, cluster-ICMP, and firewall-
relation receipts are also pinned to the exact Service, NodePool, or NodeClass
UID through issue, final authority, readback, transition, and clearance, so a
same-name Kubernetes replacement cannot inherit an in-flight mutation. Valid
RC.2 relation receipts that predate the UID companion annotation are migrated
by one exact CAS and a zero-cloud-I/O reconciliation pass; malformed or foreign
state remains fail closed.

Bootstrap, CSI, CCM, and Karpenter now share the same receipt lifecycle: when
the current CAS-owning invocation proves that a post-issue authority failure
occurred before the InSpace mutation method was called, it rejects or resets
only that exact receipt using a bounded persistence context. Once the cloud
method is called, every HTTP status, malformed response, timeout, disconnect,
or restart remains ambiguous and retains its durable lock until authoritative
readback resolves it. CSI applies this rule to post-CAS disk-create inventory,
duplicate-name, ownership, readiness, attach, and detach exits as well: caller
cancellation cannot strand a proven-undispatched Lease, while an advanced or
replacement receipt is preserved by exact-state CAS.

The shared HTTP-400 not-found matcher is restricted to InSpace's exact
`No such virtual machine exists` / `No such VM exists` response, optionally
followed by only a canonical UUID. Billing, network, authorization, and other
`No such ... exists` errors can no longer become VM-absence authority.

## RC.4 teardown and malformed-success corrections

RC.4 adds explicit 600-second CSI provisioner and attacher RPC timeouts,
allowing two minutes of preflight before a final local guard requires 480
seconds to remain for the five-minute InSpace mutation deadline, two-minute
durable destructive recovery, and one-minute final readback/Lease margin. An
insufficient deadline issues no cloud mutation and exact-clears only that
invocation's undispatched Lease. Karpenter reconciles each durable cleanup
receipt once, uses three read-only final snapshots that reject resurrection,
and reports fully converged deletion as typed `NodeClaimNotFound`, eliminating
RC.3's duplicate finalizer cycle.

All SDK list endpoints reject empty/`null` HTTP-200 bodies, `null` or empty
rows, malformed identities, and duplicates; destructive VPC reads also reject
an omitted membership list. The missing-VM HTTP-400 compatibility response is
bound to the exact VM-detail GET and requested UUID, so it cannot make a disk
or another resource appear absent. CCM exact-reads name discoveries, requires
configured-VPC membership for active VM identity, and corroborates every
providerID-backed 404, missing-VM HTTP 400, or `Deleted` tombstone with valid
VM-inventory and VPC omission before `InstanceExists` may return false. A
providerID-less list omission remains an error during external-CCM
initialization.

Bootstrap, CCM, CSI, and Karpenter validate every configured-VPC VM membership
row rather than scanning only for the target. Malformed unrelated UUIDs and
case-insensitive duplicates now block creation, attachment, deletion, target
selection, fenced discovery, and established protection audits before any
cloud mutation or absence conclusion.

Every SDK endpoint now has an explicit response contract. Redirects,
undocumented success statuses, oversized or incomplete bodies,
declared-length mismatches, trailing JSON, and duplicate JSON object keys are
non-authoritative. Floating-IP detail reads require independent complete-list
corroboration, relationship-bearing responses must explicitly carry their
assignment collections, and a firewall reconstructed from list data remains
discovery-only rather than exact absence authority.

The guarded release harness binds its clean local annotated tag object and
peeled commit to the canonical GitHub tag, verifies each linux/amd64 image
config's source/version/revision labels, checks both released chart archives
against `SHA256SUMS`, and requires byte-identical OCI chart pulls. It persists
that complete artifact set before any cloud mutation and installs the verified
local chart archives. Offline teardown accepts only the preserved
version-scoped runner whose source, image, and chart binding matches the run;
it never rebuilds old cleanup logic from a newer checkout.

The live harness also disables proxies and redirects for canonical InSpace
HTTPS reads, enforces a 4 MiB response limit and three stable account
observations, and rejects malformed or omitted identity data. Before deleting
any Kubernetes owner it fsyncs all historical cloud identities; offline
teardown consumes that journal so later discovery omissions cannot hide

## Mutation outcome safety

All 22 shared InSpace mutation methods are single-dispatch, never automatically
replayed, and cannot follow mutation redirects. Bootstrap, CCM, CSI, and
Karpenter persist exact issued receipts before creates, replacements,
relationship changes, and destructive requests, then resolve them only through
authoritative readback. No HTTP error status, malformed response, transport
failure, or restart can trigger a blind replay. A still-visible resource or
relationship never grants a second destructive dispatch. Exact duplicate
firewall and paid-NLB relationship rows are treated as one set member while
malformed or conflicting rows fail closed.

After each Kubernetes or status-store CAS, every controller mutation path
repeats its exact target or create-absence authority immediately before
dispatch. The file-journal live probe does the same after fsync.
Mutation-response UUIDs are provisional until canonical ownership readback
promotes the unique intended resource; a valid-looking foreign UUID never
becomes protection or cleanup authority.

Every bootstrap VM-absence observation is corroborated by the exact VM
endpoint, location-wide VM inventory, and configured VPC membership; any
positive or split read resets the durable evidence. Standard NLB and CSI
mutations also require canonical resource UUID, billing-account, and VPC
identity. CSI exact-reads the union of every unfiltered location VM row and
every configured-VPC member before concluding that a disk is detached. A disk
visible on a VM outside that VPC therefore blocks delete or a second attach.
Disk-delete and detach operations complete only after three Lease-persisted
absence observations at least 30 seconds apart; reappearance clears only that
evidence.

CCM uses exact owner CAS for ICMP, shard, and Service firewall mutations.
Deterministic-name creates re-list after that CAS, adopt one exact owned match,
dispatch only from exact absence, and treat response UUIDs as provisional.
Firewall relationship removal requires two durable absence observations
separated by 30 seconds, so a transient omission cannot clear its receipt and
replay a DELETE. Karpenter uses the fixed
`karpenter-inspace-firewall-mutations` Lease for independent per-firewall CAS;
only the winning invocation may assign or detach its exact base firewall, and
its RBAC cannot update unrelated Leases. Karpenter persists the issued receipt
and writes a terminal destructive/removal result only after three complete
authoritative observations at least 30 seconds apart. Reappearance resets the
sequence; restart before the terminal write starts it again without replaying
the request. VM create repeats deterministic-name, ownership, firewall,
FIP-collision, and VPC authority after NodeClaim CAS, and promotes a response
UUID only after canonical detail, billing, and exact VPC-membership proof.

Fixed bootstrap adds bounded `status.createAttempts` and
`status.deleteAttempts` ledgers to the `InSpaceCluster` CRD. CSI uses
namespace-scoped Lease receipts. Existing
installations require no manual migration. Fixed control-plane VM POSTs are
now serialized (`maxParallelControlPlaneCreates: 1`) so each VM's restrictive
firewall assignment is authoritatively proven before the next VM is created;
subsequent boots may overlap. Karpenter conservatively decodes v2
NodeClaim create fences as v3, including exact base-firewall and floating-IP
update receipts; unresolved issued v2 operations remain read-only until exact
cloud state or operator resolution is available. Its cleanup finalizer retries
unrelated NodeClaim update conflicts without repeating cloud deletion, and
fails closed if provider-owned fence state changed after the cloud audit. Other
new status and receipt objects are created as operations are reconciled.

The maintainer `make live-test` firewall probe also journals exact intent before
dispatch and never replays an ambiguous POST or DELETE. After journal fsync,
create repeats deterministic-name inventory and delete repeats exact
UUID/name/account/policy and empty-assignment authority. The create response
UUID is diagnostic only. Delete completion needs three journal-persisted
absence observations at least 30 seconds apart; restart resumes the count and
reappearance resets it. The unsafe direct multi-resource Go lifecycle targets
are retired. Stable release acceptance uses the restart-durable full-cluster
E2E workflow instead.

Release acceptance captures and compares six location-scoped resource kinds
(VMs, networks, firewalls, floating IPs, load balancers, and disks) in every
location returned by the account's authoritative location inventory, plus the
account-global bucket and service-package inventories. The local mutation

only active private network; cleanup preserves and exactly compares that
