## Hostname readback race

RC.2 live acceptance exposed an early cloud-init timing race after the exact
`127.0.1.1 <generated-hostname>` line had been appended successfully to
`/etc/hosts`. The immediate `getent` check did not yet return that mapping, so
bootstrap stopped before package preparation even though the file update had
succeeded.

RC.3 retries the exact local-hostname readback within a fixed bound on the
bastion, control-plane nodes, and Karpenter workers. Bootstrap still fails
closed when the expected `127.0.1.1` mapping never becomes visible; the retry
only tolerates the observed short NSS readback delay.

## Breaking upgrade note

The bounded fixed-node bootstrap change advances control-plane and bastion
ownership from RC.2 schema v5 to v6. Existing RC.2 fixed VMs are not adopted
in place; destroy and recreate the cluster to apply RC.3. Owned teardown
continues to recognize the supported older fixed-node records.

Karpenter bootstrap advances from RC.2 v10 to v11. Workers rendered by the
previous candidate are eligible for drift replacement through the normal
NodeClaim lifecycle.

## Validation scope

Unit and static contracts cover the bounded hostname-readback loop and the v6
fixed-node/v11 worker bootstrap markers. Destructive RC.3 live acceptance is
still required; these notes do not claim that it has passed.
