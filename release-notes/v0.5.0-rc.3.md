## Public NodeLB bootstrap floor

RC.2 live acceptance showed that the former 1-vCPU/2-GiB public NodeLB default
could start a VM and connect to the RKE2 supervisor without bringing kubelet to
Ready. RC.3 raises the shared and dedicated memory default to 4 GiB while
retaining 1 vCPU for live acceptance.

- Dedicated CPU and memory annotations now enforce at least 1 CPU and 4 GiB in
  addition to requiring an exact finite-catalog shape.
- Existing default-sized Services migrate through the controller's guarded
  shard replacement path. The exact legacy 1-vCPU/2-GiB NodePool remains
  authorized only while its owned shadow Service still advertises that shard;
  new capacity can never use the compatibility renderer. Before upgrading,
  change any dedicated Service that explicitly requests a sub-4-GiB shape;
  those requests are now rejected.
- The AMD EPYC host class and 30-GiB root disk are unchanged. The generic
  Karpenter catalog also removes the undersized 1-vCPU/2-GiB shape; its smallest
  advertised instance is now 1 vCPU/4 GiB.

Before upgrading, remove any generic NodePool requirement pinned to
`is-general-1c-2g` and let Karpenter move that capacity to `is-memory-1c-4g` or
a larger shape. Existing ownership records remain readable and deletable, but
the withdrawn type cannot launch a replacement and its existing NodeClaims may
be reported as drifted after the upgrade.

## Validation scope

Unit and static contracts cover defaulting, minimum validation, rendered
NodePool requirements, Node labels, and the InSpace VM ownership shape. Full
destructive RC.3 live acceptance is still required; these notes do not claim
that it has passed.
