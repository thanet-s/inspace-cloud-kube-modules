# v0.6.0-rc.15

RC.15 aligns the remaining mutation success contracts with responses proven
by a minimal zero-residue InSpace API probe:

- VM and firewall creation accept HTTP 200 or 201 only with complete validated
  JSON identities.
- Firewall-to-VM relationship removal accepts only HTTP 200 or 204 with
  exactly zero response body bytes.
- Floating-IP deletion accepts only HTTP 200 with the exact JSON literal
  `true`.

Each rule remains isolated to its exact route. It does not broaden parent
firewall, NLB, or any other mutation contract.

This candidate also closes the Karpenter lifecycle missed-wakeup observed
during RC.14 teardown. Eligible deleting, registered NodeClaims poll uncached
Node state every five seconds while their matching Node remains. Recovery
requires two uncached absence proofs at least 30 seconds apart, the exact
durable launch identity and removal fence, and a typed terminal
`NodeClaimNotFound` provider result. It then rechecks identity and Node absence
before CAS-removing only Karpenter's termination finalizer, leaving the
provider create-protection finalizer to complete its independent cloud audit.

Stable v0.6.0 promotion still requires this exact unchanged candidate commit to
pass the complete guarded real-cloud E2E lifecycle and zero-residue teardown.
