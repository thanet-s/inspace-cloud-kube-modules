## Live teardown and API readback corrections

RC.4 supersedes RC.3 after guarded real-cloud acceptance completed the fixed
cluster, Karpenter, CSI, private-LB, and paid-NLB exercises, then reached the
NodeLB lifecycle and exposed two teardown-duration bugs plus additional
malformed-success readback cases. Stable promotion remains gated on this exact
candidate passing the complete guarded lifecycle and restoring the isolated
account to its immutable zero-billable-resource baseline.

CSI now gives both `csi-provisioner` and `csi-attacher` an explicit 600-second
RPC timeout. This allows two minutes for preflight work, then a final local
guard requires 480 seconds to remain before CreateDisk, DeleteDisk, AttachDisk,
or DetachDisk: five minutes for the shared client's mutation deadline, two
minutes for durable destructive recovery, and one minute for final readback
and Lease persistence. An insufficient deadline issues no cloud mutation and
exact-clears only that invocation's undispatched Lease. The Helm schema rejects
shorter values; the standalone manifest and real-cluster acceptance assert the
same bound.

Karpenter fenced-create cleanup reconciles each durable VM/FIP/firewall receipt
once, then performs three read-only exact snapshots that still reject any
resource or relationship resurrection. A fully converged provider deletion
returns typed `NodeClaimNotFound`, avoiding a second complete deletion cycle.
This removes the RC.3 finalizer delay without weakening the existing no-replay
or three-observation contracts.

All nine shared SDK list methods now distinguish a valid `[]` from an empty or
JSON `null` response and reject `null`/empty elements, malformed canonical
identities, and duplicates. Destructive VPC absence also rejects an omitted or
`null` `vm_uuids` field. Bootstrap, CCM, CSI, and Karpenter therefore cannot
turn a malformed HTTP-200 collection into absence authority.

The SDK now applies an endpoint-local response contract to every read and
mutation. Redirects, undocumented success statuses, oversized or incomplete
bodies, declared-length mismatches, trailing JSON, and duplicate JSON object
keys are rejected as non-authoritative. Exact floating-IP reads are
corroborated by a complete list, and relationship-bearing responses must
explicitly contain their assignment collections before a controller may infer
presence or absence. A synthetic firewall detail reconstructed from a list is
discovery data only and can no longer prove exact absence.

Bootstrap, CCM, CSI, and Karpenter additionally canonicalize and validate every
configured-VPC VM membership row, rejecting malformed unrelated rows and
case-insensitive duplicates before creation, attachment, deletion, target
selection, fenced discovery, or protection audits. Target-only membership
scans can no longer hide contradictory rows.

The InSpace missing-VM HTTP-400 compatibility rule is now bound to the exact
VM-detail GET and its requested UUID. A missing-VM phrase returned by a disk,
network, list, or other endpoint remains an ordinary error. CCM uses VM lists
only for discovery, exact-reads the selected UUID, and proves exactly one
configured-VPC membership before publishing any active InstancesV2 result.

Kubernetes cloud-node-lifecycle deletes a NotReady Node after one
`InstanceExists=false` result. CCM therefore corroborates a providerID-backed
404, bound missing-VM HTTP 400, or canonical HTTP-200 `Deleted` tombstone with
both valid VM-inventory omission and exact configured-VPC membership omission
in the same call. Any split view, malformed identity, duplicate, read error,
or positive row fails closed. A providerID-less name omission also remains an
error during external-CCM initialization.

Release acceptance now binds the clean local annotated tag object and peeled
commit to the canonical GitHub tag. It checks every linux/amd64 product image
config label against that source, version, and revision, verifies both GitHub
chart archives against `SHA256SUMS`, and requires each OCI chart pull to be
byte-identical. The exact chart archives, source revision, image digests, and
chart checksums are durably persisted before cloud mutation; Helm installs
those local verified archives. Teardown uses a preserved version-scoped runner
whose complete binding matches that manifest and makes no GitHub or registry
request. Missing, mismatched, or newer teardown code fails closed.

The live harness uses canonical HTTPS requests with proxies and redirects
disabled, caps every response at 4 MiB, requires three stable account reads,
and rejects malformed identities or relationship omissions. It durably records
every historical cloud identity before deleting Kubernetes owners, then uses
that journal during offline teardown so a resource that disappeared from
discovery cannot be forgotten.
