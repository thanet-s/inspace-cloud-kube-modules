## Endpoint-local public Services

This candidate adds the fifth load-balancing mode,
`public-node-local`. A Service selects an operator-owned edge pool, uses
`externalTrafficPolicy: Local`, and publishes the floating IP of every Ready
node that has a Ready local endpoint. The data path preserves the real client
address and supports low TCP or UDP ports without data-port NodePorts.
Kubernetes still allocates its standard `healthCheckNodePort` for each Local
LoadBalancer Service; CCM does not expose that port through the InSpace
firewall. `publishNotReadyAddresses: true` is rejected so an unready endpoint
cannot bypass the Local readiness gate.

CCM creates an exact-owned private Cilium child Service and one canonical
per-Service InSpace firewall. It verifies the Node, VM, private address, and
assigned floating IP before publication. Karpenter-backed nodes additionally
require the exact NodeClaim, NodePool, NodeClass, and persisted VM ownership
chain with `firewallProfile: public-node-local`; administrator-managed nodes
require the protected pool label. CCM never creates, moves, or deletes edge
VMs or their floating IPs in this mode. Karpenter-backed nodes additionally
prove public-IP reservation, the configured trusted private base firewall,
and the VM's complete base-plus-Service firewall assignment set.

Protocol/port claims are exclusive across a named pool. A deterministic UID
winner prevents two Services from programming the same frontend, and stale
frontends block reuse until their firewall and private status are withdrawn.
Readiness, endpoint, identity, policy, or cloud-readback drift fails closed.

The Karpenter example uses a static AMD EPYC NodePool with an explicit replica
count, `expireAfter: Never`, zero voluntary-disruption budget, a dedicated
taint, and a one-node replacement surge. Node replacement may change the
published address because the FIP belongs to the node.

The guarded acceptance suite creates a real 1-vCPU/2-GiB edge node, verifies
the public Proxy/private VIP pair, exact TCP/80 plus UDP/443 firewall and FIP
ownership, both public protocol paths, retained HTTP client source address,
endpoint withdrawal and restoration, zero InSpace NLB
creation, a forced static-NodePool replacement with old VM/FIP cleanup and
live-Service reconvergence, Service-only cleanup, and final VM/FIP cleanup.
Full destructive acceptance against the published RC.1 artifacts is still
required; these notes do not claim that it has passed.

## Upgrade note

This is additive relative to v0.5.0. Existing private, paid InSpace NLB,
`public-node-shared`, and `public-node-dedicated` Services keep their current
contracts. No earlier stable release exposed `public-node-local`.
