# v0.6.0-rc.10

RC.9 exposed a provider-side VM-create ambiguity while provisioning control
plane slot 1. The POST was durably marked issued, returned without a usable
identity, and never appeared in authoritative VM inventory. The controller
correctly refused to replay the paid POST, but `--until-ready` had no terminal
deadline and therefore remained in read-only recovery indefinitely.

This release candidate keeps the no-replay fence and adds bounded liveness.
An issued VM create that remains authoritatively absent gets a ten-minute
restart-stable recovery window from its durable issue time. Multi-pass
`--until-ready` and `--delete` operations also have a configurable 30-minute
overall deadline. Expiry returns nonzero, preserves every mutation receipt,
and includes the last unresolved error or progress message; it never clears a
fence or dispatches a second mutation. Continuous and single-pass controller
modes remain unbounded.

The shared client now records a strictly validated
`X-Warren-Correlation-Id` from failed, truncated, malformed, redirected, or
contract-incompatible HTTP responses. The identifier is diagnostic only,
never sent as an idempotency key, and is retained in deadline errors for
InSpace support. Unsafe, duplicate, or oversized header values are discarded.
The public InSpace VM API documents no idempotency-key or non-commit retry
contract, so automatic POST replay remains intentionally unsupported.

CSI's Kubernetes Lease authority now applies the same strict response boundary
as the cloud client: one MiB plus one-byte overflow detection, declared-length
verification, one complete JSON value, and duplicate-key rejection. A
truncated prefix, trailing value, or duplicate resource-version/fence field
can no longer become mutation authority. Case variants of schema-bound fields
are also rejected because Kubernetes' JSON decoder otherwise treats, for
example, `metadata` and `Metadata` as the same field; annotation-map keys keep
their normal case-sensitive semantics.

Fixed bootstrap and Karpenter now contain a second liveness case: a VM create
can materialize successfully while its immediately following base-firewall
assignment remains ambiguously issued; Karpenter applies the same rule to its
floating-IP metadata update. The controllers continue read-only recovery for
a durable two-minute protection window. Fixed bootstrap uses this window for
an established or restart-recovered materialized VM; failure in the initiating
fresh-create call still selects immediate containment. At expiry the
controllers perform a final authoritative read, let a visible commit win, and
otherwise CAS-persist rollback before deleting the exact newly created VM and
its exact floating IP. Restart and late commit are covered; an issued POST or
PATCH is never replayed, and adopted infrastructure is never selected for
automatic rollback.

Karpenter also durably times a protection failure when a previously observed
firewall or floating-IP state later drifts. Successful readback clears that
marker; persistent drift selects the same exact rollback after two minutes.
Floating-IP cleanup accepts only the receipt-bound coherent pre-PATCH
blank/zero state or the complete desired name/account state. Partial or
foreign metadata fails closed, and neither issued nor observed recovery
replays the PATCH.

The interrupted RC.9 run was stopped through the guarded cleanup path. It
removed the bastion, control-plane slot 0, both floating IPs, and both
firewalls. The final deterministic audit performed three strict reads and
reported zero VMs, floating IPs, firewalls, load balancers, or disks; the
complete isolated-account inventory exactly matched its pre-run baseline.
