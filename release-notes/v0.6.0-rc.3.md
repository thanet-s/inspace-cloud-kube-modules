## Live-acceptance readback corrections

RC.3 supersedes RC.2 after destructive acceptance exposed sparse InSpace VM
network fields, HTTP-200 deleted-resource tombstones, and several
proven-pre-dispatch receipt lifecycle gaps. Stable promotion still requires
the complete guarded real-cloud lifecycle and exact return to the immutable
account baseline; these notes do not claim that acceptance has passed yet.

Bootstrap mutation authority, CSI attachment, CCM load-balancer target/mutation
authority, and Karpenter fenced-create/cleanup authority accept an omitted VM
`network_uuid` only when an independent configured-VPC read proves exactly one
membership for the canonical UUID. A present contradictory network, wrong
billing or identity, duplicate/missing membership, or an active-path VM with
status `Deleted` remains fail closed. CCM's read-only InstancesV2 identity
paths also reject HTTP-200 `Deleted` tombstones but do not treat a sparse
network field as VPC mutation authority.

CSI and Karpenter deletion recovery now handle canonical HTTP-200 tombstones
without weakening active use. CSI requires an exact disk UUID/billing
tombstone, `ListDisks` omission, and no attachment across the exact-read
VM/configured-VPC union for three spaced durable observations. Karpenter
requires immutable VM identity and ownership plus independent `ListVMs` and
configured-VPC omission for the same three observations. Restart resumes the
same receipt and never blindly replays the delete. CSI also clears only its
exact proven-undispatched Lease when post-CAS create/attach/detach authority
fails, using a bounded detached context; concurrent replacement receipts
survive.

Paid InSpace NLB cleanup now remains reachable through Kubernetes Service
deletion when the exact owned FIP is clean or a durable cleanup receipt exists.
NLB HTTP-200 `is_deleted` tombstones require canonical identity plus active-list
omission. Every paid-NLB and NodeLB mutation retains its receipt after any HTTP
status, timeout, disconnect, malformed success, or redirect until authoritative
readback converges; only a typed local pre-dispatch block or final authority
rejection may exact-reset the winning receipt.

NodeLB shard, cluster-ICMP, and firewall-relation lifecycles are pinned to the
exact Service, NodePool, or NodeClass UID through issue, final authority,
readback, transition, and clearance. Same-name replacements therefore receive
no stale cloud mutation. Valid RC.2 relation receipts without the new owner-UID
companion are bound by one exact CAS and a reconciliation pass with zero cloud
I/O; malformed or foreign state stays fail closed.

Karpenter cleanup reserves enough bounded time for every sequential VM,
floating-IP, and firewall-relation absence phase, preventing an earlier slow
phase from starving finalizer completion. Its create/adoption authority rejects
deleted VM tombstones and launch-shape drift before any firewall, floating-IP,
or VM mutation.

The shared HTTP-400 not-found matcher now recognizes only InSpace's exact
missing-VM phrase, optionally followed by one canonical UUID. Billing, network,
authorization, and other generic HTTP-400 errors remain ordinary failures.
Release acceptance inventories VMs, networks, firewalls, floating IPs, NLBs,
and disks in every location plus global buckets and managed-service packages;
the configured prepared VPC is preserved as the sole immutable network
baseline.
