## Attached-volume VM deletion guard

v0.6.1 prevents the Karpenter provider from deleting a VM while any
non-primary block volume remains attached. Normal termination and
provider-owned rollback both require the canonical VM storage inventory to
contain exactly one primary root disk and no non-primary entries before
deletion authorization. The provider repeats that exact check immediately
before the VM DELETE.

An attached volume, omitted inventory, or ambiguous primary layout now fails
closed before dispatch. The error remains retryable, the NodeClaim stays
terminating, and the VM, Floating IP, and firewall assignments remain intact.
After CSI detaches every additional volume, reconciliation can continue through
the existing fenced deletion and repeated absence proofs. A post-authorization
attachment or storage-identity drift rejects only the current undispatched
receipt.

Deleting a VM directly through the InSpace dashboard or API bypasses this
provider guard and can delete every attached non-primary block volume and its
data. Important volumes still require independent backups. The InSpace API has
no conditional VM DELETE, so the final exact storage read minimizes but cannot
eliminate a race with an out-of-band attachment.

This patch changes no CRD or Helm values. Release acceptance covers a blocked
Karpenter deletion with an attached RWO volume, CSI detachment, worker
replacement, reattachment of the same disk UUID, and persisted-data readback.
