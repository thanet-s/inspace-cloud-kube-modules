## Public NodeLB private-VIP datapath

This is an intentional RC-only contract reset. RC.4 does not adopt or migrate
the earlier RC.1-RC.3 Cilium Node IPAM datapath or child identities; destroy and
recreate any pre-RC.4 NodeLB test cluster. No stable release exposed that path.

RC.4 creates one exact-owned, same-namespace
`inlb-dp-<service-identity>` child Service for every public NodeLB Service. The
identity is the first 52 lowercase hex characters of SHA-256 over
`namespace NUL name NUL Service-UID`, is repeated in the
`inspace.cloud/node-lb-service-id` label, and is bound by an exact controller
owner reference. The generated Service uses
`loadBalancerClass: inspace.cloud/node-datapath`, disables NodePorts, and
publishes each selected Node's private InternalIP with `ipMode: VIP`. The
user-facing `inspace.cloud/node` Service publishes the paired InSpace FIP with
`ipMode: Proxy`. Reconciliation and live acceptance require the FIP's
`assigned_to_private_ip` to equal the Node InternalIP and private VIP exactly.

This matches the real InSpace packet path: its floating-IP DNAT rewrites the
destination to the VM private address before Cilium sees the packet. Cilium now
programs only that private low-port frontend, so TCP/80, TCP/443, and UDP/443 do
not depend on NodePorts and the parent and generated Services cannot compete for
one public frontend. `Service.spec.externalIPs` is not used.
The Cilium Node IPAM controller remains disabled and is not part of this path.

The parent records
`service.inspace.cloud/node-lb-datapath-active-shard` before any private VIP.
Activation then follows one ordered gate: exact private VIP, persisted firewall
assignment fence, exact Service-firewall assignment readback, and public Proxy
status. Removal detaches and reads back the public edge before either status is
shrunk. Missing, terminating, foreign, or drifted children fail closed; they are
never adopted.

Migration prepares replacement capacity while the old path is active, but the
fixed child identity makes final cutover deliberately break-before-make. An
ambiguous firewall assignment retains its marker/private VIP for a five-minute
grace and three consecutive empty readbacks; a late assignment resets that
proof. Emergency withdrawal can still detach an exact persisted firewall after
rule drift without weakening normal policy ownership.

## Node registration correction

NodeLB identity labels under
`inspace.cloud.node-restriction.kubernetes.io/*` remain on NodePools and
NodeClaims, but are no longer emitted as kubelet bootstrap labels. Kubernetes
correctly rejects kubelet attempts to set that protected namespace. Karpenter
applies the labels through the API after registration, preserving the
anti-forgery boundary. The earlier 1-vCPU/4-GiB registration failure was caused
by this label restriction, not memory exhaustion; RC.4 keeps 1 vCPU/4 GiB as
the default until a smaller shape is separately live-proven.

## Control-plane VIP convergence

New control-plane VMs use bootstrap ownership schema v8 and configure kube-vip
with a 500 ms gratuitous-ARP interval plus 5 s/3 s/1 s leader-election lease,
renew, and retry timings. Static and live acceptance verify those exact values;
teardown continues to recognize every previously supported bootstrap schema.

## Release acceptance

The guarded live suite requires exact public `Proxy` and private `VIP` status,
the canonical digest name, internal datapath class, exact ownership and active
marker, zero NodePorts, public TCP/80 and TCP/443 responses, public UDP/443
response, and no Cilium duplicate-frontend log. Partial deletion and final
teardown must remove datapaths, NodePools, NodeClaims, VMs, FIPs, firewalls, and
any accidental InSpace NLB before the account baseline is accepted.
