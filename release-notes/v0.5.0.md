## Public node load balancers

v0.5.0 introduces Karpenter-backed public NodeLB Services alongside the
existing InSpace NLB and private Cilium L2 modes. Shared mode reuses capacity
when Service port/protocol tuples do not conflict and creates another shard
when they do; dedicated mode gives one Service its own shard. The CCM creates
the managed NodeClass and NodePools, one mutable aggregate public firewall per
shard, and the shared ICMP firewall. NodeLB nodes are AMD EPYC, tainted against
application workloads, and default to 1 vCPU, 2 GiB RAM, and a 30 GiB root
disk.

Each public Service has an exact-owned same-namespace datapath Service. InSpace
maps the public floating IP to the node's private address, while Cilium exposes
that private address as a VIP. This supports low ports and mixed protocols,
including TCP/80, TCP/443, and UDP/443, without NodePorts. The parent reports
the public floating IP with `ipMode: Proxy`; the generated datapath reports the
private node address with `ipMode: VIP`.

Activation and removal are ordered fail-closed transitions. CCM first proves
the generated child spec with an empty private-VIP status, then commits and
reads back the aggregate policy and assignment before publishing the private
VIP or public status. Each shard
firewall has the stable name
`inlb-<cluster-ownership-hash>-shard-<shard-hash>` and keeps the same UUID while
CCM updates its canonical TCP/UDP rule union in place with `PUT`. The full
SHA-256 applied policy hash and crash-recovery fence are stored separately on
the shard NodePool. A CCM finalizer keeps that state anchor alive through
NodePool deletion, and exact cloud readback gates every activation. NodePool
deletion uses foreground propagation so owned NodeClaims terminate while the
CCM finalizer still retains the aggregate-firewall ledger; this avoids a
finalizer/garbage-collection circular wait. CCM also upgrades an already-started
background deletion to an exact UID-fenced foreground request while managed
NodeClaims remain, without re-adding the foreground finalizer after they drain.
The generated NodeClass has an independent
`inspace.cloud/node-lb-cluster-state` finalizer for the shared ICMP transaction
ledger. External deletion performs a fail-closed cluster drain; last-owner
cleanup proves cloud absence and persists a Service handoff before the
NodeClass can disappear.

Every NodeLB VM has exactly its private base firewall and shared cluster ICMP
firewall, plus at most one aggregate shard firewall. Adding a compatible shared
Service stages its private child, expands the aggregate, and publishes only the
new public status; sibling VM, FIP, NodePool, firewall UUID, status, and traffic
identities remain unchanged. Removing a member first narrows the aggregate and
then removes only that Service's statuses and child. A NotReady node is
withdrawn from readiness and status without detaching the aggregate, avoiding
a detach/reattach feedback loop. The final Service and finalized NodePool
anchor cleanup until shard capacity is gone and three spaced authoritative
reads prove the unassigned aggregate firewall is deleted.
An exact NodePool/finalizer readback is handed off durably on every referencing
Service; missing, drifted, or marker-only shard state now fails closed instead
of being reassigned while paid capacity may remain. Firewall attachment for a
new surge/replacement VM no longer flaps established shards or shared siblings;
the new VM stays unadvertised until a later authoritative readback.
Aggregate and shared-ICMP firewall creates now have a durable one-POST fence.
An ambiguous paid create is never reissued merely because repeated list calls
are empty; the owning finalizer remains until the original exact resource is
observable or the attempt is manually resolved after cloud-side proof.
Ambiguous aggregate-policy PUTs also retain their issued fence without a timed
retry, preventing an older delayed request from overwriting a later policy
generation.

## Karpenter VM lifecycle hardening

Karpenter VM creation now has its own Kubernetes-durable one-POST fence and
`karpenter.inspace.cloud/create-protection` finalizer. The exact SDK response
UUID is persisted before firewall, Floating-IP, or ProviderID work, and an
independent rollback decision cannot race materialization. Adoption baselines
retain the exact target-to-Floating-IP association so VM deletion cannot erase
dependent correlation by auto-unassigning an address.

Deletion and rollback now use a staged VM-first contract: dispatch the exact
UUID delete, prove core Get/List/VPC absence, remove only the correlated
Floating IP, prove VM and assignment absence again, then detach firewalls. A
UUID-less ambiguous POST is never reissued or released merely because lists are
empty. The finalizer may therefore remain permanently until an exact VM appears
or an operator uses the documented issue-bound resolution after complete
read-only inventory and, for `no-result`, InSpace support confirms terminal
no-commit. Blind fence JSON edits or finalizer removal are unsafe and
unsupported.

After a firewall assignment mutation, the provider now gives an exact duplicate
of the intended assignment the existing bounded readback window to converge.
The VM is already protected during that wait. Persistent duplicates still roll
back, while established duplicates, foreign firewalls, malformed resources,
and invalid NodeLB shard or ICMP policies continue to fail closed immediately.

## Node and bootstrap updates

- Karpenter's catalog includes CPU and memory requirement labels and InSpace
  pricing based on requested CPU and RAM. The catalog includes
  `is-general-1c-2g` for the NodeLB default.
- Protected NodeLB identity labels are applied through the Kubernetes API
  after registration instead of being sent as forbidden kubelet labels.
- Fixed-node kube-vip convergence uses faster ARP and leader-election timing.
- The guarded release environment can skip the package upgrade step while
  retaining mirror and node configuration, reducing disposable test bootstrap
  time.

## Upgrade note

No earlier stable release exposed the public NodeLB API. Clusters created with
v0.5.0 RC.1-RC.3 used the retired Cilium Node IPAM experiment and must be
destroyed and recreated. RC.4 and RC.5 used the final private-VIP datapath but
an RC-only per-Service firewall scheme that is not adopted by the stable
release. Recreate every release-candidate test cluster for clean acceptance.
